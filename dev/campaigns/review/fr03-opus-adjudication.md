### F-01 — validate_rules `--manifest` completeness check fails OPEN off-canonical-path
**Verdict: REAL. Severity OVERSTATED (major → minor). Recommend a pre-tag 2-line fix, but it is NOT a tag-blocker.**

The defect is real and I confirmed it by reading the code, not just trusting the repro. `_missing_manifest_entries` (`validate_rules.py:311-317`) derives `rules_dir = manifest_path.resolve(strict=False).parent / "kotlin"` and, if that directory is absent, `return []` — a silent "no missing entries." That is a genuine fail-open in the dangerous direction (silent pass, not confusing fail), and FR-02's two-direction repro is sound.

Why **not major**:
- **No production path is exposed at this tip.** The task asked me to test the "sole caller" premise; FR-02 is imprecise here. There are **two** callers of `--manifest`: `ci-gate.sh:62` and `test_kotlin_antipattern_policy.py:185`. I read both. Both pass the **canonical** `.claude/hooks/rules/manifest.json`, whose parent has the sibling `kotlin/` dir — so both hit the correct fail-closed branch. FR-02's direction-2 repro (with the sibling present, removal → exit 1) proves the enforcing path works. The test caller additionally only asserts the positive direction (`returncode == 0`), so it is not a false-green either.
- **The gate this protects is a meta-gate** (does a `kotlin/*.yaml` rule have a manifest entry). Even if it no-ops, the actual architecture enforcement still runs on the 64 rules that *are* in the manifest. A newly-added uncovered rule is still caught by the canonical per-commit `ci-gate.sh` run. For the fail-open to bite, you need a future invocation that points `--manifest` at an off-canonical copy — a plausible future refactor or a naively-written missing-entry regression test, but nothing does it today.

Why **still worth a pre-tag fix** (not info):
- It fails **open**, and it is *precisely the campaign's signature bug class* — GH-19 exists because "validate_rules was unwired for weeks" (ledger, GH-19 title). A fail-open inside `validate_rules` is the same silent-no-op failure the whole campaign was built to kill.
- **The fix is 3 lines away in the same file.** `new_entry_mode` (:361) and the default rules_dir (:377) both already anchor to `Path(__file__).resolve().parent / "kotlin"`, and `detect-nonintegrated-kotlin.py:77` uses the identical robust pattern (FR-02 hold #23). The completeness check is the lone hold-out. Anchoring it to `Path(__file__)` (or erroring instead of returning `[]`) is a trivial, in-idiom change.

**Reachability:** latent, plausibly-reachable-by-future-change, inert at this tip. **Land pre-tag** because it is ~2 lines and belongs to the gate-hardening campaign it is part of; **but tagging without it harms no production path today.** This is the finding stage 4 should re-probe hardest (see §D).
