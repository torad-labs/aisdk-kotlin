## 1. [HIGH] AnthropicProvider.kt:1043 — Streaming thinking `signature_delta` (and `citations_delta`) is treated as a fatal stream error, aborting generation
**Evidence:** In `AnthropicStreamState.accept`, the `content_block_delta` handler (lines 1034-1044) only recognizes `text_delta`, `thinking_delta`, and `input_json_delta`. The `else` branch at line 1043 returns `StreamEvent.Error("...unsupported content_block_delta type `$deltaType`.")` for any other subtype. But the Anthropic Messages API emits a `signature_delta` content_block_delta for EVERY thinking block in streaming mode — it follows the `thinking_delta` events and carries the block's cryptographic signature — and a `citations_delta` for cited text blocks when document citations are enabled. Both fall through to line 1043. `StreamEvent.Error` is fatal downstream: `ToolLoopAgent.kt:436` (`is StreamEvent.Error -> throw UiMessageStreamError(...)`) and `StreamObjectResult.kt:26/60/79` throw on it. Thinking is a first-class supported path in this provider — the request builder sets `thinking` (lines 657-666) and the stream parser emits `ReasoningStart/Delta/End` — and `AnthropicAwsProvider` delegates to this same model, so both providers are affected. Concretely: a streamed `claude-*` request with extended/adaptive thinking enabled produces `thinking_delta` (handled) then `signature_delta` (→ fatal error), aborting the whole generation the instant the signature arrives. The streaming tests (AnthropicProviderTest.kt lines 450-510, 544-575) only exercise the three handled subtypes; the test at line 629 covers an unknown block INDEX, not an unknown delta TYPE — there is no `signature_delta` coverage. Secondary effect: even a raw `Flow<StreamEvent>` consumer never receives the signature, so the streamed reasoning block cannot be replayed in a follow-up turn (`anthropicAssistantPart` reads `metadata["signature"]` at lines 897-902, which the non-streaming path populates at lines 285-287 but the streaming path never does).
**Fix:** In the `content_block_delta` `when` block, (1) add a `"signature_delta"` case that captures `delta["signature"]` onto the `AnthropicStreamBlock` and surfaces it on the eventual `StreamEvent.ReasoningEnd` providerMetadata (mirroring the non-streaming `signature` capture at lines 285-287), so streamed reasoning carries the signature for replay; (2) add a `"citations_delta"` case (emit a Source or accumulate the citation, matching the non-streaming citation path); and (3) change the `else` branch (line 1043) from returning a fatal `StreamEvent.Error` to a forward-compatible no-op (`return emptyList()` / `{}`), matching upstream Vercel AI SDK which ignores unknown delta subtypes. Keep the `null` case (line 1042) as an error, since a missing `delta.type` is genuinely malformed.

## 2. [HIGH] BedrockEventStream.kt:93 — Mid-stream Bedrock modeled exceptions are silently dropped (`:exception-type` header never read)
**Evidence:** decode() derives the event key as `headers[":event-type"].orEmpty().ifBlank { headers[":error-code"].orEmpty().replaceFirstChar { lowercase } }` (lines 93-95). Per the AWS Smithy event-stream spec (smithy.io/2.0/aws/amazon-eventstream.html), the four exceptions that BedrockStreamState.accept checks for — internalServerException, modelStreamErrorException, throttlingException, validationException — are MODELED error events, framed as `:message-type: exception` + `:exception-type: <unionMemberName>` (camelCase, e.g. "throttlingException"). The `:error-code` header is ONLY used for UNMODELED errors (`:message-type: error`). So when Bedrock emits a real mid-stream throttle/internal-server/validation error: `:event-type` is absent (blank) and `:error-code` is absent (blank) -> eventType resolves to "". messagePayload (line 40-41) sees the blank eventType and returns the raw payload `{"message":"..."}` unwrapped. BedrockStreamState.accept (BedrockResponse.kt:171) then evaluates `value["internalServerException"] ?: value["throttlingException"] ...` against `{"message":"..."}`, finds nothing, error stays null, and no StreamEvent.Error is emitted. The stream simply ends (channel closes); finish() (BedrockResponse.kt:255) reports the default finishReason=Other (messageStop never arrives on an exception frame). Net effect: a throttled/failed streaming generation is presented to the consumer as a silently truncated success with no error — the exact failure the accept() error-check was written to surface.
**Fix:** Read `:exception-type` before falling back to `:error-code`, and do NOT lowercase the first char of `:exception-type` (it is already the camelCase union member name matching accept()'s keys). E.g.: `eventType = headers[":event-type"].orEmpty().ifBlank { headers[":exception-type"].orEmpty() }.ifBlank { headers[":error-code"].orEmpty().replaceFirstChar { it.lowercaseChar() } }`.

## 16. [MEDIUM] MCP.kt:1110 — HTTP transport: POST that returns text/event-stream blocks send() until the stream closes, hanging the request
**Evidence:** In HttpMCPTransport.postMessage, when a request POST returns Content-Type: text/event-stream, the response is consumed synchronously: `McpSseFrame.parseStreamReleasing(response.bodyAsChannel()) { event -> ... onMessage?.invoke(...) }` (line 1110) suspends in its readLine() loop until the server closes the stream. send() therefore does not return until EOF. Concrete path: a tool call goes callTool -> request("tools/call", options=MCPRequestOptions(signal=abortSignal)); since neither timeoutMillis nor maxTotalTimeoutMillis is set, request() calls requestWithoutTimeout WITHOUT any withRealTimeout wrapper. Inside, `transport.send(...)` is awaited BEFORE `deferred.await()`. The MCP Streamable-HTTP spec permits a server to deliver the JSON-RPC response on the POST's own SSE stream and keep that stream open (e.g. to interleave server-initiated messages) rather than closing immediately. When that happens, parseStreamReleasing never returns, send() never returns, and although onMessage already completed the deferred, `deferred.await()` on the next line is unreachable — the tool call hangs forever. The Vercel reference (mcp-http-transport.ts, content-type text/event-stream branch) deliberately calls `processEvents(); return;` WITHOUT awaiting, so send() returns immediately and the response is delivered by the background reader.
**Fix:** Do not await the POST-response SSE stream inside send(). Launch the parse on the transport's connection scope (lifecycle.scopeOrNull()) the same way readInboundSse() is launched, then return from postMessage immediately — mirroring the reference's non-blocking `processEvents(); return;`. The deferred is still completed by the background reader, and deferred.await() (optionally bounded by a timeout) governs request completion.

## 21. [LOW] MCP.kt:657 — JSON-RPC response correlation is type-strict, dropping responses whose id is echoed with a different JSON type
**Evidence:** rpcIdKey() (lines 657-664) encodes the JSON type into the key: a number id becomes "n:<content>" and a string id becomes "s:<content>". Outbound requests always use `JsonPrimitive(requestMessageId++)` (line 493), a JSON number, so they register under "n:0", "n:1", etc. In onResponse (lines 622-642) the lookup is `responseHandlers[rpcIdKey(response.id)]`. If a server echoes the numeric request id as a JSON string (e.g. returns "id":"0" instead of 0), rpcIdKey yields "s:0" which never matches the registered "n:0"; the handler is not found, onResponse throws "Protocol error: ... unknown message ID" (routed to onError by the readers), the deferred is never completed, and the request hangs (or times out, e.g. the 30s initialize handshake). **Fix:** Normalize the id to its content for lookup the way the reference coerces with Number(): since this client only ever issues numeric ids, key the responseHandlers map by the primitive's content string (dropping the s:/n: type discriminator), or in onResponse fall back to the numeric-keyed entry when the typed lookup misses, so a server echoing a numeric id as a string still resolves its handler.
