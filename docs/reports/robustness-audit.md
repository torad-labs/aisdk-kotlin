### [W1 · HIGH] Parallel tool execution deadlocks permanently when a tool throws AbortError/CancellationException (the designed abort path black-holes)
`src/commonMain/kotlin/ai/torad/aisdk/ToolLoopAgent.kt:1150` — agent-loop/concurrency-cancellation

**Trigger:** A step runs >=1 non-gated tool concurrently in the `coroutineScope { ... }` parallel block. While a child is executing, any CancellationException escapes its `executeTool` call WITHOUT the parent coroutine being structurally cancelled. The two realistic triggers: (1) the host calls AbortController.abort() and an abort-aware tool executor calls `ctx.abortSignal.throwIfAborted()` (the first-class cancellation pattern in this codebase, AbortSignal.kt:95) — this throws AbortError, which extends CancellationException (AbortSignal.kt:124); (2) a tool executor internally uses `withTimeout {}` and the timeout fires (TimeoutCancellationException) and escapes the executor.

**Defect:** In the bounded-parallel tool block, each tool runs in a child `launch` (line 1088) and reports back via exactly one `signals.send(ParallelToolSignal.Completed(...))` (line 1138). The consumer loop terminates purely by counting completions: `while (completedChildren < toolsToExecute.size) { signals.receive() }` (line 1150). `executeTool` correctly rethrows CancellationException (lines 1352-1356), and AbortError IS a CancellationException. When a child coroutine completes exceptionally with a CancellationException, Kotlin structured concurrency treats it as that child cancelling ITSELF: `JobSupport.childCancelled` returns true for a CancellationException without cancelling the parent, so the surrounding `coroutineScope` is NOT cancelled and its sibling/consumer coroutines keep running. But that child never reached its `signals.send(Completed(...))`, so `completedChildren` can never reach `toolsToExecute.size`. The consumer's `signals.receive()` suspends forever. Because AbortSignal in this codebase is deliberately decoupled from coroutine cancellation (DECISION 3 — abort() cancels an internal SupervisorJob and fires callbacks, never the agent coroutine), the consumer coroutine is itself NOT cancelled, so it cannot unwind. The entire `coroutineScope` (and thus `streamInternal`, `generate()`/`stream()`/the engine loop) hangs indefinitely and leaks the coroutine. This is the opposite of the intended abort behavior: the step-boundary abort handling (lines 681-685: emit Abort, fireAbort, return) and the model-stream AbortError handling (lines 912-920) are completely bypassed for the parallel-tool phase — pressing stop while tools run in parallel makes the agent hang forever instead of aborting. Note structural cancellation of the parent (e.g. host cancels the scope, or withTimeout around generate) is fine, because then the consumer's receive() is itself a cancellation point; only the codebase's own AbortSignal path — the path users are told to use — deadlocks.

**Fix:** Do not terminate by counting Completed signals from children that can legitimately disappear via CancellationException. Two robust options: (A) Use `async` per tool + `awaitAll()` (draining progress concurrently): a child that completes with a CancellationException then has its Deferred re-throw it in the awaiting parent, which cancels the `coroutineScope` and propagates the AbortError up to the existing AbortError handling — converting the hang into a clean abort. (B) Wrap each child body in try/finally that ALWAYS sends a terminal signal even on cancellation — `try { ... } catch (ce: CancellationException) { withContext(NonCancellable) { signals.send(Completed(...marking aborted...)) }; throw ce }` — and additionally have the consumer cooperate with the abort (poll abortSignal or use select) so the step can emit StreamEvent.Abort + fireAbort and return instead of waiting on the dead child. Option A is the idiomatic structured-concurrency fix and restores correct abort propagation.

### [W1 · HIGH] Tool-failure messages fed back to the model are silently mis-flagged as successful (isError=false)
`src/commonMain/kotlin/ai/torad/aisdk/ToolLoopAgent.kt:1517` — agent-loop/error-handling-silent-failure

**Trigger:** Any tool failure that goes through emitToolErrorDeferred: tool executor throws, NoSuchTool, invalid/undecodable tool input, repair failure, approval-categorization failure, or an approval-gated tool whose input only decoded after repair. The loop then continues to the next step (multi-step agent), re-sending the accumulated messages to the model.

**Defect:** emitToolErrorDeferred builds the persisted tool-result message via `ToolMessage(toolCallId, toolName, JsonPrimitive(msg))`. The `ToolMessage` factory (ModelMessage.kt:39) creates `ContentPart.ToolResult(toolCallId, toolName, output)` with `isError` defaulting to `false` (ModelMessage.kt:153). So while the live `StreamEvent.ToolError` correctly carries `error=...` for observers, the message appended to the conversation log records the failure as a NON-error tool result whose body is just the error string. On the next loop step this message is re-sent to the provider, and the providers map this flag to the wire error signal: AnthropicProvider.kt:806 `if (result.isError) put("is_error", JsonPrimitive(true))` and GoogleInteractionsModel.kt:531. Because isError is false, `is_error` is never emitted — the model is told the failed tool SUCCEEDED and returned the error text as a normal result, which suppresses the model's error-aware self-correction behavior and makes the failure indistinguishable from success on persistence/rehydration. This is inconsistent with every sibling path that handles the same semantic correctly: applyToolResult Success carries result.isError (ToolLoopAgent.kt:1464), ToolApprovalCoordinator.applyDenied sets isError=true (ToolApprovalCoordinator.kt:264), and AgentSession.submitStreaming sets isError=true for ToolError events (AgentSession.kt:205). The same bug exists at ToolApprovalCoordinator.kt:232 (emitToolError in the approval-resume path), where an approved tool that fails is likewise re-logged with isError=false.

**Fix:** Add an `isError` parameter to the `ToolMessage` factory (or construct ContentPart.ToolResult directly) and pass `isError = true` for every tool-error path: emitToolErrorDeferred (ToolLoopAgent.kt:1517) and ToolApprovalCoordinator.emitToolError (ToolApprovalCoordinator.kt:232). Ideally also wrap the error text in a structured error output (mirroring applyDenied's ToolResultOutput.ExecutionDenied) so modelVisible/isError are consistent. This makes the failure flow to the model with is_error=true and survive persistence as an error.

### [W1 · HIGH] Loop terminates prematurely when every tool call in a step fails categorization (decode/repair/NoSuchTool), suppressing model self-correction
`src/commonMain/kotlin/ai/torad/aisdk/ToolLoopAgent.kt:1226` — agent-loop/edge-cases-input-robustness

**Trigger:** In one step the model emits one or more tool calls and ALL of them fail the categorization pass - e.g. a single malformed-JSON tool argument that repair cannot fix, or a hallucinated tool name (NoSuchTool). Each failed call goes to deferredToolErrorMessages, none reach toolsToExecute.

**Defect:** hasLocalToolWork (line 1035) is computed as toolsRequiringApproval.isNotEmpty() || toolsToExecute.isNotEmpty(). Calls that fail to decode/repair or have no matching tool are appended to messages as tool-error results but are NOT counted as local tool work. The natural-termination check at line 1226 (`if (!hasLocalToolWork && effectiveFinishReason != ToolApprovalRequested) break`) then ends the loop even though the provider's finishReason is tool-calls and fresh tool-error results were just written for the model to react to. This is asymmetric with a tool that DECODES fine but THROWS at execution time: that path goes through toolsToExecute -> hasLocalToolWork == true -> the loop continues and the model sees the error and can retry. So a model that emits malformed args gets zero retry, while a model that emits valid args to a failing tool gets retries. A single malformed tool call can silently end the conversation.

**Fix:** Count emitted tool-error results as work that should continue the loop: include `deferredToolErrorMessages.isNotEmpty()` (or a flag set whenever a tool-error result was appended this step) in hasLocalToolWork, so the model gets re-invoked to react to its malformed call instead of the loop terminating.

### [W2 · HIGH] A single malformed/unknown-ID inbound message permanently kills the SSE/HTTP reader (and tears down the whole client)
`src/commonMain/kotlin/ai/torad/aisdk/MCP.kt:1145` — mcp/concurrency-cancellation

**Trigger:** The server sends one inbound SSE frame that either (a) fails JSON-RPC parsing (JSONRPCMessage.fromJson throws), or (b) is a response whose id is unknown to the client — e.g. a duplicate response, or a late response for a request whose handler was already removed after a timeout/abort. onResponse() then throws MCPClientError("...unknown message ID").

**Defect:** In HttpMCPTransport.readInboundSse (line 1145-1149) and SseMCPTransport.start's reader (line 1229-1242), the onMessage callback is invoked directly inside McpSseFrame.parseStream, which propagates any exception from the callback. onMessage -> onResponse throws on an unknown id (MCP.kt:614-620 / 622-634), and JSONRPCMessage.fromJson throws on any malformed frame. That exception unwinds out of parseStream into the reader's catch block, which (correctly) rethrows CancellationException but for every OTHER throwable just calls onError and lets the reader loop EXIT. For HTTP the inbound SSE stream dies (only restarting if inboundRetryRequested was set); for SSE the finally fires the client's onClose (line 1257-1260), which sets isClosed and fails every pending request with "Connection closed" — so one bad frame permanently closes the entire MCP client. This contrasts with Experimental_StdioMCPTransport (MCP.kt:1464-1469), which deliberately wraps each onMessage in a per-message try/catch (rethrowing CancellationException, routing other errors to onError) so a single bad line cannot terminate the read loop. The SSE/HTTP readers lack that isolation.

**Fix:** Isolate per-message handling exactly like the stdio reader does: wrap the `onMessage?.invoke(JSONRPCMessage.fromJson(event.data))` in a try/catch INSIDE the parseStream lambda, rethrow CancellationException, and route all other throwables to onError without letting them escape parseStream. (Optionally also make onResponse for an unknown id route to onError instead of throwing.) That keeps the reader alive across malformed/late/duplicate messages, matching both the stdio transport and the TS SDK behavior.

### [W2 · HIGH] Stray/late/duplicate JSON-RPC response ID tears down the SSE reader and (for SseMCPTransport) the entire connection
`/home/marcos/Documents/dev/opensource/aisdk-kotlin/src/commonMain/kotlin/ai/torad/aisdk/MCP.kt:1240` — mcp/error-handling-silent-failure

**Trigger:** A JSON-RPC response/error arrives on the inbound SSE stream whose id is not (or no longer) in responseHandlers — e.g. a response that arrives after its request timed out via withRealTimeout, a duplicate response, or a response for a request the server is slow to answer. onResponse() throws MCPClientError("Protocol error: Received a response for an unknown message ID").

**Defect:** onResponse(JSONRPCResponse) and onResponse(JSONRPCError) (MCP.kt:614-634) THROW when the response id is missing from responseHandlers. In the background SSE readers this exception escapes the per-event dispatch and is not contained per-message. In SseMCPTransport.start (line 1240) `"message" -> onMessage?.invoke(JSONRPCMessage.fromJson(event.data))` runs inside parseStream with no per-message guard; the throw unwinds out of parseStream into the reader's catch (line 1243) and then the finally (line 1257) calls lifecycle.onReaderExited() and fires onClose() — which drives DefaultMCPClient.onClose() -> failAllPendingRequests(), failing EVERY in-flight request with "Connection closed". ### [W2 · HIGH] A single late/duplicate/malformed inbound message aborts the entire SSE inbound stream and tears down the connection
`src/commonMain/kotlin/ai/torad/aisdk/MCP.kt:1240` — mcp/edge-cases-input-robustness

**Trigger:** Any exception thrown while dispatching one inbound SSE event causes the whole reader to die. Concrete sequences: (a) a request times out via withRealTimeout (request() line 469) so requestWithoutTimeout's finally removes its handler (line 505); the server later delivers that response over the SSE stream -> onResponse finds no handler and throws MCPClientError 'unknown message ID' (lines 615-618 / 622-625). (b) A buggy server sends a duplicate response for an id (first completes+removes handler, second is 'unknown'). (c) A 'message' event whose data is empty/whitespace/not-JSON makes JSONRPCMessage.fromJson -> WireDecoder.parseObject throw. In all cases the throw propagates out of McpSseFrame.parseStream (which has no per-event try/catch) into the reader's catch.

**Defect:** McpSseFrame.parseStream (lines 1388-1400) invokes the onEvent callback with no exception isolation. In SseMCPTransport.start the callback at line 1240 calls onMessage?.invoke(JSONRPCMessage.fromJson(event.data)); onMessage -> DefaultMCPClient.onMessage -> onResponse, which throws MCPClientError for an unknown id, and fromJson throws on malformed data. Because parseStream does not catch, the exception unwinds into the reader's catch (line 1243): since `established` is true it fires onError, then the finally (lines 1257-1260) calls lifecycle.onReaderExited(), cancels the scope and fires onClose. onClose -> DefaultMCPClient.onClose -> failAllPendingRequests(), so every other in-flight request is failed and the MCP session is permanently dead. The HttpMCPTransport.readInboundSse path (lines 1145-1149) has the same exposure: one bad frame ends the inbound stream and loses all subsequent server->client messages/notifications in that stream. This is demonstrably the wrong behavior because the stdio reader (lines 1464-1469) wraps each onMessage dispatch in a try/catch that routes non-cancellation errors to onError and continues the loop — proving per-message isolation is the intended contract, just not applied to the SSE/HTTP transports. A late response after a timeout is an ordinary, expected occurrence, so this turns a benign event into a full connection loss.

**Fix:** Isolate per-message dispatch in the SSE/HTTP readers exactly as the stdio reader does: wrap `onMessage?.invoke(JSONRPCMessage.fromJson(event.data))` (lines 1240 and 1146) in try { } catch (e: CancellationException) { throw e } catch (e: Throwable) { onError?.invoke(e) } so a single bad/late/duplicate message is reported but does not unwind parseStream. Additionally, onResponse for an unknown id should route to onError (or be ignored) rather than throw, since a late response after timeout is expected and must not kill the stream.

### [W2 · HIGH] HMAC-SHA256 crashes (IndexOutOfBoundsException) for any key longer than 64 bytes — breaks tool-approval signing with a long secret
`src/commonMain/kotlin/ai/torad/aisdk/Crypto.kt:91` — tools-approval/edge-cases-input-robustness

**Trigger:** Configure ToolLoopAgent.experimental_toolApprovalSecret (a raw ByteArray) with more than 64 bytes — e.g. a passphrase like "my-very-long-approval-secret-passphrase-used-across-the-fleet-2026".encodeToByteArray() (65+ bytes). The first approval-gated tool call reaches ToolApprovalSignature.signToolApproval -> CryptoPrimitives.hmacSha256(secret, ...) and throws instead of producing a signature. The same crash hits verifyToolApprovalSignature on the resume path.

**Defect:** Standard HMAC requires that a key longer than the block size be hashed AND then zero-padded back to the block size. Here the `key.size > blockSize` branch returns `sha256(key)`, a 32-byte array, with no padding. The `else`/`<` branches yield a full 64-byte `normalizedKey`, but the >64 branch yields only 32 bytes. The following loop runs `for (index in 0 until blockSize)` (0..63) and reads `normalizedKey[index]`, so at index 32 it throws IndexOutOfBoundsException. The tool-approval secret is arbitrary user-supplied bytes, so secrets over 64 bytes are entirely plausible and turn every gated tool call into a hard crash. (Existing callers — SigV4 derived keys, KlingAI JWT — happen to stay <=64 bytes, which is why this has gone unnoticed; tests only exercise a 15-byte secret.)

**Fix:** Zero-pad the hashed key back to the block size: `key.size > blockSize -> sha256(key) + ByteArray(blockSize - 32)` (or run the hashed key through the same `+ ByteArray(blockSize - key.size)` padding as the short-key branch). Add a unit test with a >64-byte key to lock the contract.

### [W2 · HIGH] Spawned MCP stdio child's stderr is never drained, redirected, or closed — child blocks and FD leaks
`src/jvmAndAndroidMain/kotlin/ai/torad/aisdk/MCPStdioProcess.jvmAndAndroid.kt:17` — json-wire/resource-safety

**Trigger:** An MCP server launched via Experimental_StdioMCPTransport writes more than the OS stderr pipe-buffer capacity (~64 KB on Linux) to its stderr — common for servers that log diagnostics/startup banners to stderr. Because ProcessBuilder defaults stderr to Redirect.PIPE and nothing in the parent ever reads process.errorStream, the child blocks on its next stderr write. It then stops servicing stdout, so reader.readLine() never returns, every in-flight MCP request hangs until its handshake/request timeout, and the connection is effectively dead.

**Defect:** JvmMCPStdioProcess starts the process with a bare ProcessBuilder (only stdin/stdout are wired via inputStream/outputStream). ProcessBuilder leaves stderr as Redirect.PIPE by default, which creates an OS pipe that the parent is responsible for draining. The parent never reads process.errorStream, so the pipe buffer fills and the child process blocks (classic ProcessBuilder deadlock). Separately, close() closes the writer and reader but never closes process.errorStream, leaking that file descriptor for the lifetime of the JVM until GC finalizes the Process. Note: the naive fix redirectErrorStream(true) is WRONG here because it would interleave stderr text into the stdout channel that carries the line-delimited JSON-RPC messages, corrupting fromJson parsing.

**Fix:** On the ProcessBuilder, set redirectError to ProcessBuilder.Redirect.DISCARD (or .INHERIT) so the child's stderr is consumed by the OS / parent terminal and can never block — do NOT use redirectErrorStream(true), which would corrupt the JSON-RPC stdout stream. If stderr must be captured, drain process.errorStream on a dedicated background coroutine/thread for the process lifetime. Also add runCatching { process.errorStream.close() } in close() to release the FD.

### [W3 · MEDIUM] EventStreamParser silently yields an empty, error-free stream when the body contains no SSE data lines
`src/commonMain/kotlin/ai/torad/aisdk/EventStreamParser.kt:29` — streaming-sse/error-handling-silent-failure

**Trigger:** A 2xx response whose body is NOT SSE-framed — e.g. a proxy/gateway returns HTTP 200 with a plain JSON error envelope ({"error":...}) or an HTML interstitial instead of text/event-stream. streamSse() only throws on non-2xx status, so the body is passed to parse() as-is.

**Defect:** The line loop only accumulates lines that startsWith("data:") and flushes on blank lines; every other line is silently discarded. If no line starts with "data:", eventData stays empty, flush() returns early, and the Flow completes having emitted zero ParseResult values — no Success, no Failure. Downstream (forwardSseEvents / OpenAICompatible stream) then emits only ResponseMetadata + a synthetic Finish, so the caller receives an empty, apparently-successful assistant turn. A real upstream error delivered with a 200 status and non-SSE framing is completely swallowed instead of surfaced as a ParseResult.Failure / StreamEvent.Error.

**Fix:** After the stream completes, if no event was ever emitted AND the buffer/body contained non-blank content that was not SSE-framed, emit a single ParseResult.Failure carrying the raw body so the consumer can surface it (e.g. track 'emittedAny' and the leftover non-data text, then emit Failure when emittedAny is false and content was seen). Alternatively validate content-type is text/event-stream in streamSse before treating the body as SSE.

### [W3 · MEDIUM] fromOpenAI crashes the whole stream on non-primitive id/model/created fields
`src/commonMain/kotlin/ai/torad/aisdk/Streaming.kt:95` — streaming-sse/edge-cases-input-robustness

**Trigger:** An OpenAI-compatible SSE chunk where id, model, or created is a JSON object/array instead of a primitive, e.g. `data: {"id":{"x":1},"choices":[...]}` or `data: {"created":[1]}`. kotlinx's `JsonElement.jsonPrimitive` throws IllegalArgumentException when the element is not a JsonPrimitive.

**Defect:** `fromOpenAI` does `obj["id"]?.jsonPrimitive?.contentOrNull` (and the same for `model` and `created`). `?.jsonPrimitive` only guards null — if the key is present but maps to a JsonObject/JsonArray, the `jsonPrimitive` accessor throws IllegalArgumentException. This is reached from every OpenAI-compatible stream collector (e.g. OpenAICompatibleModels.kt:234) inside a `channelFlow { ... collect { } }` block that has NO try/catch around it. So one malformed/quirky metadata field on a single chunk does not degrade to a graceful `StreamEvent.Error` — it propagates out of the flow and aborts the entire stream collection with an unhandled exception. The same applies to `created` where `.jsonPrimitive` is evaluated before `doubleOrNull`.

**Fix:** Use a non-throwing cast for each field, e.g. `val id = (obj["id"] as? JsonPrimitive)?.contentOrNull` (and likewise for model/created). `as?` returns null for non-primitive shapes instead of throwing, preserving the intended 'absent metadata -> null -> return null' robustness contract.

### [W3 · MEDIUM] Pending-request handler leaks when a request is cancelled/times out (suspend cleanup in finally runs in a cancelled coroutine)
`src/commonMain/kotlin/ai/torad/aisdk/MCP.kt:504` — mcp/concurrency-cancellation

**Trigger:** A request is cancelled while awaiting its response — e.g. options.timeoutMillis elapses (withRealTimeout cancels the block), the AbortSignal aborts, or the caller's scope is cancelled — AND the responseHandlersMutex is momentarily contended (another coroutine in onResponse / failAllPendingRequests / a concurrent request holds it).

**Defect:** requestWithoutTimeout registers a CompletableDeferred in responseHandlers under the mutex, then in its finally block removes it via `responseHandlersMutex.withLock { responseHandlers.remove(...) }`. When the coroutine has already been cancelled (timeout/abort/scope-cancel), this is a suspending call on a cancelled coroutine: Mutex.lock() takes its non-suspending fast path only when uncontended; if it has to actually suspend (contended), it observes the cancellation and throws CancellationException before acquiring the lock, so responseHandlers.remove(...) never runs. The deferred is never completed (await just threw) and the entry stays in the map. Each such cancellation can strand an entry; the table only gets cleared at close() (failAllPendingRequests). **Fix:** Run the cleanup under NonCancellable so it cannot be skipped by the in-flight cancellation: `} finally { withContext(NonCancellable) { responseHandlersMutex.withLock { responseHandlers.remove(rpcIdKey(id)) } } }`.

### [W3 · MEDIUM] fixJson drops INSIDE_STRING_UNICODE_ESCAPE state — truncation mid-\uXXXX yields invalid JSON, fails repair
`/home/marcos/Documents/dev/opensource/aisdk-kotlin/src/commonMain/kotlin/ai/torad/aisdk/FixJson.kt:262` — json-wire/edge-cases-input-robustness

**Trigger:** A streaming/partial JSON string value cut in the middle of a \uXXXX unicode escape — e.g. the wire delivers {"name":"caf\u00 (truncated after \u or after 1–3 of the 4 hex digits). Because raw parse fails on truncated input, parsePartialJson falls through to fixJson, which is the only path that hits this state.

**Defect:** The Vercel v6 fix-json scanner has two escape states: INSIDE_STRING_ESCAPE and INSIDE_STRING_UNICODE_ESCAPE (with a unicodeEscapeDigits counter). When it sees \u it enters the unicode-escape state and only advances lastValidIndex once all 4 hex digits have arrived; if the stream is cut before that, lastValidIndex stays before the \u, the drain appends a closing '"', and the partial escape is cleanly discarded → valid JSON → RepairedParse.

The Kotlin port omits INSIDE_STRING_UNICODE_ESCAPE entirely (the enum at lines 28-44 has no such variant) and its INSIDE_STRING_ESCAPE handler unconditionally pops and sets lastValidIndex = i for ANY escaped char, including 'u'. It therefore treats \u as a complete escape and then treats the trailing hex digits as ordinary string chars, each advancing lastValidIndex. On truncation mid-escape, lastValidIndex points inside the incomplete escape; the drain appends '"', producing invalid JSON such as "caf\u00" or "abc\u". kotlinx parseToJsonElement throws, tryParseJson returns null, and parsePartialJson returns FailedParse instead of RepairedParse. Downstream, ExtractJsonMiddleware.extractAndRepairJson then returns the unrepaired region (it only re-serializes on RepairedParse), so Output.decode fails on a chunk that upstream would have repaired. Note: complete escapes and non-truncated input are unaffected (those bytes round-trip identically and the raw-parse-first path handles complete values), so this is strictly a truncated-mid-escape robustness regression, not silent corruption.

**Fix:** Add an INSIDE_STRING_UNICODE_ESCAPE variant to FixJsonState plus a unicodeEscapeDigits counter (var, like literalStart). In the INSIDE_STRING_ESCAPE branch, pop then: if char == 'u' reset the counter and push INSIDE_STRING_UNICODE_ESCAPE WITHOUT advancing lastValidIndex; else set lastValidIndex = i (current behavior). Add an INSIDE_STRING_UNICODE_ESCAPE branch in the main loop that, on a hex digit, increments the counter and only on the 4th digit pops the frame and sets lastValidIndex = i (non-hex chars are ignored, no advance). In drainClose, INSIDE_STRING_UNICODE_ESCAPE contributes "" (the partial escape is simply excluded by lastValidIndex). This mirrors v6 exactly.

### [W3 · MEDIUM] Data race on plain MutableList in CombineAbortSignals teardown vs. wiring loop
`src/commonMain/kotlin/ai/torad/aisdk/AbortSignal.kt:159` — core-transport/concurrency-cancellation

**Trigger:** Combine two or more not-yet-aborted signals; while the wiring `for` loop is still executing `registrations += registration`, one of the source signals fires from a DIFFERENT thread (e.g. a UI stop button on thread B while combination runs on thread A).
