## Phase 3 â€” Leverage (Opus)

LEVERAGE POINT: Producer-side seam contracts are declared as defaulted optional fields ahead of (or entirely without) the upstream stage ever populating them. The codebase ships a "type surface now, wire the producer later" convention â€” public event/result/wire fields are added with `= null` / `= emptyList()` defaults, constructed positionally so the defaults silently win, and then locked in by tests that assert the empty/null state. The KDoc admits it verbatim at Lifecycle.kt:31-33 ("Loop-side population is staged in as a follow-up; the type surface is in place now") and Lifecycle.kt:39-42 ("Null when the loop hasn't yet wired the field"). The single missing mechanism is producer-obligation enforcement: nothing forces the stage that emits a type to supply every declared field from data it already has in scope.

IF CHANGED: Every consumer-facing field becomes either populated-by-construction or absent-by-design â€” no third "declared but permanently zero-state" category. Making the new fields required parameters of the producer-facing constructor (or routing all construction through a single factory that takes the in-scope data) turns each of these from a silent runtime emptiness into a compile error at the one construction site, so the data that already exists nearby (completedSteps at ToolLoopAgent.kt:605, callParams at 689, streamResult.request, the explicit isError flag) gets threaded instead of dropped. The "assert it's empty" contract tests flip to "assert it round-trips," removing the cement.

IF NOT CHANGED: The list of phantom fields keeps growing with every parity gap. Each new event/result/wire type added "for surface compatibility" ships another null/empty field that observers, telemetry, and cross-language clients read as real data and get zero. Tests written against the zero-state harden each one in place, so the bugs are not just unfixed but actively defended. New providers and external fixtures that omit a defaulted flag silently inherit the wrong default (Streaming.kt:242). The public API accumulates trust-eroding fields that look usable but never are.
