TypeScript and JavaScript Pattern Review

Codebase: aisdk-kotlin. Generated on June 20, 2026 after six parallel read-only review lanes: API/modeling, nullability/type safety, singletons/global state, async/error handling, tests/docs/tooling, and structural concentration.

Executive Summary

The repo has made real Kotlin moves: sealed stream/content events, cold Flow APIs, explicit API mode, ABI validation, kotlinx.serialization, and value classes for some IDs. The remaining TS/JS residue is concentrated in compatibility surfaces, provider facades, tool/UI state, and generated parity tooling.

The highest leverage move is to draw a hard boundary between Kotlin-native SDK APIs and compatibility/wire adapters. Right now the compatibility layer is not just present; it is shaping the stable ABI, CI gates, tests, and public docs.

Triage Order

1. Fix Behavior Risks First

Cancellation handling, parallel tool streaming, semaphore placement, and provider stream parser identity fallbacks are runtime correctness issues.

2. Establish API Boundaries

Move framework facades, generated parity names, and raw provider tool shapes behind compat/wire packages before freezing 1.0 ABI.

3. Replace Optional-Prop Models

Convert nullable state bags into sealed request/state/outcome types and explicit override patches.

4. Split Barrel Modules

Break MCP.kt, ToolLoopAgent.kt, provider mega-files, and Util.kt by Kotlin responsibility.

Findings

1. Public generation APIs mirror TS optional options objects

High
Evidence
Generate.kt:20, Generate.kt:57, Generate.kt:196, Agent.kt:75.
Pattern
prompt: String? = null plus messages: List<ModelMessage> = emptyList(), then runtime require(prompt != null || messages.isNotEmpty()).
Why TS/JS
This is the Kotlin spelling of { prompt?: string, messages?: Message[] }. The required "prompt or messages" invariant is not in the type system.
Replacement
Make TextGenerationRequest or a sealed GenerationInput primary: PromptText, MessageHistory, and optionally a NonEmptyMessages value type. Keep long overloads as deprecated compatibility shims.

2. ToolLoopAgent constructor is a flattened config object

High
Evidence
ToolLoopAgent.kt:61 through ToolLoopAgent.kt:144 mixes model, instructions, tools, output, prepare hooks, sampler settings, lifecycle callbacks, approval secret, telemetry, logging, and engine context.
Pattern
One public initializer carries 40+ configuration knobs, many nullable.
Why TS/JS
This is an object-literal config shape flattened into a Kotlin constructor. It makes subclassing noisy and hides domain boundaries.
Replacement
Add ToolLoopAgent.Config or a builder DSL with grouped types: AgentIdentity, CallSettings, AgentHooks, ApprovalSettings, TelemetrySettings, and EngineSettings.

3. Lifecycle extension points erase strong types

High
Evidence
Lifecycle.kt:14, Lifecycle.kt:56, Lifecycle.kt:119, Lifecycle.kt:131, Agent.kt:122.
Pattern
options: Any?, finalOutput: Any?, experimental_context: Any?, StepResult.experimentalContext: Any?, and outputJson/errorMessage as two nullable fields.
Why TS/JS
Hook payloads act like JS event objects. Kotlin callers lose the same TContext/TOutput safety the rest of the agent API advertises.
Replacement
Make hook/event types generic where application code consumes them: AgentHooks<TContext, TOutput>, OnStartEvent<TContext>, OnFinishEvent<TContext, TOutput>. Model tool finish as Success(outputJson) or Failure(error).

4. UI tool state is a flattened discriminated union

High
Evidence
UIMessagePart.kt:72, UIMessagePart.kt:199, ConvertToModelMessages.kt:223.
Pattern
ToolUI and DynamicToolUI expose state plus nullable input, output, error, approvalId, and signature. Conversion falls back to JsonNull.
Why TS/JS
TS discriminated unions have been collapsed into { state, input?, output?, error? }. Invalid combinations are representable.
Replacement
Use sealed state variants carrying required data: InputStreaming, InputAvailable(input), ApprovalRequested(input, approvalId, signature), ApprovalResponded, OutputAvailable(input, output, preliminary), OutputDenied(reason), OutputError(error).

5. Raw provider option bags leak across public APIs

High
Evidence
LanguageModel.kt:72, KotlinApi.kt:21, Tool.kt:97, OpenAIProvider.kt:143, GoogleProvider.kt:45, AmazonBedrockProvider.kt:47.
Pattern
providerOptions: Map<String, JsonElement>, metadata: Map<String, JsonElement>, and many provider option aliases to JsonObject.
Why TS/JS
This is Record<string, unknown> preserved in Kotlin. Provider-specific settings become string keys and raw JSON instead of typed values.
Replacement
Introduce a ProviderOptions value/sealed hierarchy with typed provider entries such as ProviderOptions.OpenAI, ProviderOptions.Google, and an explicit ProviderOptions.Raw(provider, json) escape hatch.

6. Hosted provider tools are raw JSON tools

High
Evidence
OpenAIProvider.kt:143, OpenAIProvider.kt:156, GoogleProvider.kt:151, Gateway.kt:369.
Pattern
Public hosted tools are Tool<JsonElement, JsonElement, Any?>; factories accept args: JsonElement.
Why TS/JS
Provider-hosted tool inputs are dynamic object bags rather than provider-specific Kotlin data models.
Replacement
Add typed args models and builders: for example OpenAIFileSearchArgs, OpenAICodeInterpreterArgs, GoogleSearchToolArgs. Keep rawProviderTool for features not yet modeled.

7. Public provider singletons are JS module default exports in disguise

High
Evidence
OpenAIProvider.kt:46 requires HttpClient, but OpenAIProvider.kt:56 exports openai that throws at OpenAIProvider.kt:75. Same shape appears in GoogleProvider.kt:112, AmazonBedrockProvider.kt:102, OpenAICompatibleProviderFacades.kt:72, and many provider facade files.
Pattern
Top-level provider vals that cannot be used without runtime failure, plus callable provider operators such as provider(modelId).
Why TS/JS
They preserve npm package ergonomics: import { openai } from .... Kotlin DI expects explicit construction with dependencies.
Replacement
Deprecate top-level provider vals. Prefer createOpenAI(client, settings) and explicit .languageModel(ModelId). If a namespace is useful, make it stateless and non-Provider.

8. Some globals are mutable process state

Medium
Evidence
Telemetry.kt:146 exports globalTelemetry; Telemetry.kt:148 mutates it; Gateway.kt:254 exports gateway; Gateway.kt:261 caches metadata.
Pattern
Library behavior changes through process-wide registries or default service instances.
Why TS/JS
Startup-time global registration is common in JS SDKs. Kotlin libraries should be explicit and test-isolated.
Replacement
Inject telemetry through agent/provider settings. Require configured gateway transport. Keep app-wide registries in host apps, not the SDK core.

9. Parallel tool execution uses Promise.all-style buffering

High
Evidence
ToolLoopAgent.kt:940, ToolLoopAgent.kt:956, ToolLoopAgent.kt:985, ToolLoopAgent.kt:987, ToolLoopAgent.kt:1826.
Pattern
Every tool launches via async, writes to BufferingCollector, then awaitAll() completes before buffered events replay.
Why TS/JS
This matches eager Promise.all fan-out plus arrays. It weakens Kotlin Flow progress/backpressure semantics.
Replacement
Use channelFlow or a bounded worker Channel to stream preliminary events as they arrive while applying final durable message results in call order.
Verification
A fast streaming tool should emit preliminary output before a slower sibling finishes. With maxParallelToolCalls = 1, the second tool's start hook should not fire until the first finishes.

10. Public executeTool buffers a whole flow before emitting

Medium
Evidence
Tool.kt:841 through Tool.kt:858.
Pattern
Collects every tool output into mutableListOf, then emits preliminaries and final after upstream completion.
Why TS/JS
Async iterable collected into an array before returning results.
Replacement
Implement one-step lookahead in the returned Flow: emit the previous value as preliminary once the next value arrives, then emit final when upstream completes.

11. Cancellation is converted into UI errors in several paths

High
Evidence
Streams.kt:109, Chat.kt:174, CompletionApi.kt:88.
Pattern
Broad catch (Throwable) maps failures into UI error messages or error states. CompletionApi only handles AbortError separately.
Why TS/JS
Catch-all onError callback semantics. Kotlin cancellation is control flow and must be rethrown.
Replacement
Catch CancellationException first and rethrow. Model host-initiated abort as Cancelled/Aborted, not Error.

12. Provider stream parsers fabricate missing tool identities

High
Evidence
OpenResponsesProvider.kt:936, OpenResponsesProvider.kt:953, OpenResponsesProvider.kt:954. The async review also flagged AnthropicProvider.kt:843.
Pattern
Missing item_id or tool name falls back to ""; missing call id can generate a new id.
Why TS/JS
Mutable pending objects with optional properties are completed by sentinel values. Kotlin can represent protocol states explicitly.
Replacement
Parse into sealed, non-null states. Missing required ids/names should emit a typed protocol error and never produce ToolCall("", ...).

13. Settings use nullable fields as partial patches

Medium
Evidence
Context.kt:36, Context.kt:85, ToolLoopAgent.kt:604, KotlinApi.kt:21.
Pattern
AgentSettings, StepSettings, and CallSettings use null to mean "not supplied". Some fields cannot be intentionally cleared.
Why TS/JS
This is Partial<Settings> semantics. Kotlin has no undefined, so null conflates absent and clear.
Replacement
Use an explicit patch type: Override.Unchanged, Override.Set(value), Override.Clear. At minimum apply it to context, tools, system, messages, and provider options.

14. Media payloads encode unions with nullable fields and sentinels

Medium
Evidence
MediaModels.kt:8, MediaModels.kt:75, MediaModels.kt:188, ModelMessage.kt:184.
Pattern
File/image payloads use combinations of base64, url, mediaType?, and empty-string defaults.
Why TS/JS
Equivalent to { base64?: string, url?: string }. It permits both missing and double-present payloads.
Replacement
Introduce FilePayload.Base64, FilePayload.Bytes, and FilePayload.Url. Require exactly one payload variant and map to flat wire models at serialization boundaries.

15. Web framework facades are first-class KMP surface

Medium
Evidence
react/React.kt:30, react/React.kt:75, vue/Vue.kt:27, svelte/Svelte.kt:25, angular/Angular.kt:5.
Pattern
useChat, useCompletion, experimental_useObject, framework package names, and mutable context maps sit in common source.
Why TS/JS
React/Vue/Svelte/Angular hook vocabulary is being preserved as Kotlin API instead of being treated as porting compatibility.
Replacement
Lead with Kotlin surfaces: Chat, ChatSession, StateFlow, Flow, and host UI adapters. Quarantine web-shaped names under a compat artifact or explicitly experimental package excluded from stable ABI.

16. Parity tooling and CI preserve npm package topology

High
Evidence
.github/workflows/ci.yml:37, .github/workflows/ci.yml:43, .github/workflows/ci.yml:45, tools/check-ai-sdk-reference.mjs:13, tools/generate-parity-ledger.mjs:76.
Pattern
CI clones the Vercel AI SDK reference, checks npm latest, and verifies generated package ledgers before Gradle.
Why TS/JS
Kotlin evolution is blocked by upstream npm package/export drift. The compatibility matrix derives from TS package topology, not Kotlin capability boundaries.
Replacement
Split this into an advisory or scheduled drift job. Replace generated package-export ledgers with a curated matrix: Kotlin API, wire-compatible, compat-only, not applicable.

17. ABI validation is freezing compatibility names

High
Evidence
build.gradle.kts:34 states the ABI dump is the supported 1.0 contract. API dumps include compatibility names such as generateObject, streamObject, useChat, and experimental_useObject.
Pattern
Explicit API plus ABI validation is correct, but it currently includes TS-shaped facades.
Why TS/JS
Deprecated and experimental compat APIs become binary promises before the Kotlin-native shape is settled.
Replacement
Move compatibility facades to a compat artifact/package, or keep them experimental and excluded from the stable 1.0 contract until the Kotlin-native API is settled.

18. Model IDs are mostly typealias String

Low/Medium
Evidence
Gateway.kt:9, GoogleProvider.kt:39, AmazonBedrockProvider.kt:43, OpenAICompatibleProviderFacades.kt:102. Scan found 85 public typealias ... = String declarations.
Pattern
Aliases look like distinct model-id types but are assignable as plain strings.
Why TS/JS
TS type aliases can document literal unions; Kotlin typealiases add no type separation.
Replacement
Use @JvmInline value class wrappers where type separation matters, or a sealed known/custom model id hierarchy for stable provider APIs.

19. Tool execution recovers types through runtime erasure

High
Evidence
ToolLoopAgent.kt:819, ToolLoopAgent.kt:1255, Tool.kt:765, Tool.kt:772.
Pattern
Resolved tools are stored as Pair<Tool<*, *, TContext>, Any?>, later cast to Tool<Any?, Any?, TContext> and serializer types. safeValidateTypes does an unchecked cast.
Why TS/JS
This is runtime unknown recovery after schema validation. Some erasure is inevitable for heterogeneous tool sets, but it should be isolated.
Replacement
Create an internal ExecutableTool<TContext> or DecodedToolCall abstraction that captures typed decode, approval, execute, and encode operations. Constrain TInput : Any and TOutput : Any unless null payloads are intentional.

20. Engine state is flag-shaped instead of phase-shaped

Medium
Evidence
ToolLoopAgentEngine.kt:19.
Pattern
isStreaming, isModelLoading, error: String?, pendingApprovals, and lastFinishReason? can encode impossible combinations.
Why TS/JS
React-style boolean flags plus nullable slots instead of a finite state machine.
Replacement
Use sealed phases: Ready, Running, AwaitingApproval, Failed(AgentError), Cancelled, ModelLoading. Keep derived booleans as computed properties if UI wants them.

21. Large files retain TS barrel/module organization

High

Responsibility concentration is not cosmetic here. Security-sensitive auth, streaming protocol state, public API models, HTTP details, provider transforms, and tests are often grouped by upstream package name rather than Kotlin ownership.

Patterns That Should Stay

Not every singleton or raw JSON use is a problem. These were explicitly treated as acceptable:

  • Keep Sealed data object variants such as ToolChoice.Auto and StreamEvent.Abort.
  • Keep Null/no-op objects such as NoopLogger, NoopTelemetryTracer, and NoopToolStreamWriter.
  • Keep Internal cohesive utility singletons such as WireDecoder.
  • Keep JsonElement at real wire boundaries, dynamic MCP/catalog tools, and explicit raw escape hatches.
  • Keep Cold Flow as the public streaming primitive, but preserve streaming semantics instead of buffering it into lists.

Review Notes

  • This was a read-only review except for creating this report file.
  • No tests were run because the task was analysis and documentation, not behavior changes.
  • The strongest repeated signal across agents was not "raw JSON exists"; it was "raw JSON is too often the normal public path instead of an explicit boundary escape hatch."
  • The next engineering decision should be whether TS compatibility remains in the root artifact or moves to a separate compat layer before API freeze.
